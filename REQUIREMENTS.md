## 1. Project Overview

### 1.1 Purpose

ChemTabExtract is a maintained fork of `tabledataextractor` — a Python library
that extracts and standardises structured data from scientific tables using the
MIPS (*Minimum Indexing Point Search*) algorithm (Embley et al.,
DOI: 10.1007/s10032-016-0259-1). The fork is owned and maintained by Cambridge
Molecular Engineering; it will not track upstream. It is intended to replace
the `tabledataextractor` dependency in ChemDataExtractor 2 (CDE2).

### 1.2 Current State

The existing codebase is a functional Python library at version 1.5.11. Its
core logic (~2700 lines of source) is organised in a flat package layout under
`tabledataextractor/` with three sub-packages: `table/`, `input/`, and
`output/`. A test suite of ~1300 lines (11 files) covers the main algorithms,
input/output paths, and table structure. Estimated test coverage is ~40%.

Key structural observations:
- `table/algorithms.py` (1064 lines) contains all algorithmic logic in a single
  file covering MIPS search, pre-processing, header extension, SymPy
  categorisation, and table splitting.
- `input/from_any.py` imports Django solely for URL validation — the only use
  of Django anywhere in the project.
- `tabledataextractor/__init__.py` configures a `FileHandler` that writes
  `tde_log.txt` into the caller's working directory — a library anti-pattern.
- `pyproject.toml` currently specifies `requires-python = ">=3.14"` (Python
  3.14 is still in alpha as of this writing); the correct target is 3.13.
- `selenium` is a hard dependency used only as a fallback for JS-rendered URLs
  in `input/from_html.py`.
- A `build_category_table` function exists in both `table/algorithms.py` (dead
  — noted in source comments) and `output/to_pandas.py` (live). The dead copy
  should be removed.
- `setup.py` is present alongside `pyproject.toml` — legacy artefact from the
  pre-migration state.

### 1.3 Motivation for This Work

The team requires a stable, internally-maintained version of this library that:
(a) can be published under the new `chemtabextract` name, (b) sheds a heavy
transitive dependency (Django), (c) conforms to current Python packaging
standards used across the team's other projects, and (d) is covered by a
pre-commit quality gate consistent with the CDE2 project. This work also
establishes the foundation for gradual quality improvements (coverage, type
hints, complexity reduction) in future iterations.

### 1.4 Success Criteria

- The library is importable as `chemtabextract` and all public API (`Table`,
  `TrivialTable`) behaves identically to `tabledataextractor` 1.5.11.
- Django is no longer a dependency.
- `uv` manages the project; `pyproject.toml` is the sole build configuration.
- All pre-commit hooks pass on a clean checkout.
- Test coverage is measured, baseline is recorded, and the coverage gate is
  enforced at ≥ 40% (current baseline), ready to be ratcheted upward.
- CDE2 can declare `chemtabextract` as a dependency and drop `tabledataextractor`.

### 1.5 Scope Boundaries

**In scope for this iteration:**
- Rename package and all internal references from `tabledataextractor` to
  `chemtabextract`.
- Migrate to `src/` layout.
- Set `requires-python = ">=3.13"`.
- Remove `setup.py`; `pyproject.toml` + `uv` is the sole build system.
- Remove Django; replace URL validation with stdlib `urllib.parse`.
- Make `selenium` an optional dependency (`[project.optional-dependencies]`).
- Configure pre-commit following the CDE2 approach (see Section 5.1).
- Fix the hardcoded `FileHandler` logging anti-pattern in `__init__.py`.
- Split `table/algorithms.py` into four focused sub-modules.
- Remove the dead `build_category_table` from the algorithms layer.
- Establish a measured coverage baseline and enforce it via pre-commit.

**Out of scope for this iteration:**
- Any changes to public API (`Table`, `TrivialTable`, their parameters, and
  their properties/methods).
- Upstream synchronisation (no tracking of original `tabledataextractor`).
- Full type annotation of the codebase (gradual; mypy starts permissive).
- Pushing coverage above 40% (that is a subsequent, incremental effort).
- Changes to `output/to_pandas.py`, `table/footnotes.py`, `table/parse.py`,
  or `table/history.py` beyond what is required by the rename and restructure.
- Documentation site rebuild.
- Any new features.

**No-touch zones:** The algorithmic logic inside `find_cc1_cc2` (261 lines)
and the MIPS algorithm generally MUST NOT be semantically altered. Restructuring
into sub-modules is permitted; the logic itself is not.

## 3. Functional Requirements

### 3.1 Core Features (MUST)

- **MUST** expose `Table` and `TrivialTable` as the top-level public API,
  importable as `from chemtabextract import Table, TrivialTable`.
- **MUST** preserve all existing input paths: Python list, CSV file, HTML file,
  and URL (via `requests`; `selenium` optional).
- **MUST** preserve all existing output paths: CSV, pandas DataFrame, and
  pretty-print to console.
- **MUST** validate URLs without Django; the replacement MUST accept the same
  HTTP/HTTPS/FTP URL forms that Django's `URLValidator` accepted in practice.
- **MUST** not write any files to the caller's working directory on import or
  on normal operation. Library logging MUST follow the `logging.getLogger`
  pattern with no root-level handlers configured.

### 3.2 Extended Features (SHOULD / COULD)

- **SHOULD** split `table/algorithms.py` into four sub-modules
  (`utils`, `mips`, `categorize`, `structure`) with a clean `__init__.py`
  re-export so that existing import sites in `table.py` require no change.
- **SHOULD** add type hints incrementally, starting with function signatures
  in `utils.py` and public API classes, without breaking mypy in permissive
  mode.
- **COULD** add a `py.typed` marker file once meaningful type coverage exists.

### 3.3 Explicitly Out of Scope (WON'T)

- WON'T change the MIPS algorithm logic.
- WON'T add new table-extraction features.
- WON'T replace `selenium` with a modern alternative (e.g. `playwright`) in
  this iteration — that is deferred.
- WON'T add async support.
- WON'T publish to PyPI in this iteration (internal use first).

### 3.4 Data Requirements

No persistent data storage. The library operates in-memory on numpy arrays.
No changes to data model.

### 3.5 Integration Requirements

- **chemtabextract** MUST be declarable as a dependency in CDE2's
  `pyproject.toml` in place of `tabledataextractor>=1.5.11`.
- The package MUST be installable via `uv add chemtabextract` (or equivalent
  local path / VCS reference during the transition period).

## 5. Constraints

### 5.1 Technology Constraints

- **Python**: `>=3.13`; 3.13 is the preferred and tested version.
- **Build system**: `uv` + `pyproject.toml` only; `setup.py` to be removed.
- **Layout**: `src/` layout (i.e. `src/chemtabextract/`).
- **Pre-commit stack** (following CDE2 approach exactly, adapted for this repo):
  - `ruff` — lint (`--fix`, `--exit-non-zero-on-fix`) and format
  - `mypy` — permissive initially; scoped to full `src/chemtabextract/` but
    with errors suppressed via `[[tool.mypy.overrides]]` for the algorithmic
    core until type hints are added
  - `pre-commit-hooks` — trailing whitespace, EOF fixer, YAML check, large
    files, merge conflict markers, debug statements
  - `bandit` — security scan, configured via `pyproject.toml`, excludes `tests/`
  - `vulture` — dead code detection; initial whitelist to be established
  - `xenon` — complexity gate; initial thresholds to be calibrated against
    current codebase (algorithms are complex by design — thresholds MUST not
    block the initial commit)
  - `pytest-cov` — runs full test suite with `--cov-fail-under=40` as the
    initial gate; threshold is ratcheted upward in subsequent PRs
- **Pinned pre-commit hook revisions** MUST use the frozen-hash style
  (`# frozen: vX.Y.Z`) matching the CDE2 convention.
- **No new mandatory runtime dependencies** may be introduced to replace Django.
  URL validation MUST use the Python stdlib only.
- `numpy < 2.0.0` constraint is inherited and MUST be preserved until
  explicitly lifted.

### 5.2 Infrastructure Constraints

No CI/CD pipeline is specified for this iteration. Pre-commit hooks serve as
the local quality gate. A GitHub Actions workflow is not in scope but SHOULD
be added in a subsequent iteration.

### 5.3 Team & Skill Constraints

Team is experienced with Python, uv, and the CDE2 pre-commit stack. No gaps
identified. The MIPS algorithm is specialised; changes to its logic MUST be
reviewed by someone familiar with the Embley et al. paper.

### 5.4 Budget & Licensing Constraints

All dependencies MUST be open-source compatible with MIT. No new paid services
or dependencies.

## 8. Glossary

| Term | Definition |
|------|-----------|
| MIPS | Minimum Indexing Point Search — the core algorithm for locating header/data boundaries in a table, as described by Embley et al. |
| CC1–CC4 | Critical Cells 1–4 — the four coordinate pairs that define header and data regions in the MIPS algorithm |
| Stub header | The intersection region between row and column headers in a categorised table |
| Category table | The normalised output of the MIPS algorithm: each data cell paired with its full row and column header path |
| CDE2 | ChemDataExtractor 2 — the primary downstream consumer of this library |
| src layout | Python packaging convention where source code lives under `src/<package>/` to prevent accidental imports without installation |
| Ratchet | The practice of only ever increasing a coverage threshold — never decreasing it — enforced via the pre-commit hook |
| uv | Astral's fast Python package and project manager, used in place of pip/setuptools for this project |
